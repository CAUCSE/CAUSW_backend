# 🔐 환경변수(.env) 및 형상관리 가이드

## 1. 개요 및 변경 배경
2026년 5월, 인프라 개편을 통해 민감 정보의 하드코딩을 제거하고, Docker 친화적인 **단일 `.env` 체제**로 전환했습니다.

이제 `application*.yml` 설정 파일들은 Git으로 형상 관리가 되며, 실제 DB 비밀번호나 API 키 같은 민감 정보는 모두 `${ENV_VAR}` 형태로 `.env` 파일과 GitHub Secrets에서 주입받습니다.

## 2. 🚀 로컬 환경 최초 셋업 가이드 (팀원 필수)
본 레포지토리를 처음 클론(Clone) 받거나, 환경변수 개편 후 처음 Pull을 받았다면 아래 순서대로 세팅해 주세요.

1. 프로젝트 루트 경로(`app-main` 바깥 최상단)에 있는 `.env.example` 파일을 복사하여 `.env` 라는 이름의 파일을 생성합니다.
2. Notion 공유 문서 이동: [ICT 위원회 노션 ➡️ 전체 문서 DB ➡️ Github Secret 정보 ➡️ `.env (local용)`]
3. 상태가 '사용 중'인 가장 최신 버전의 내용을 복사하여 방금 만든 `.env`에 붙여넣습니다.
4. **주의:** 로컬 DB 정보(Username, Password 등)는 본인 PC 환경에 맞게 직접 수정해야 합니다. (`dev`나 `prod`용 설정을 복사하지 않도록 주의하세요!)

---

## 3. ➕ 환경변수 추가/수정 프로세스 (기능 개발 시)
개발 중 새로운 외부 API 키나 민감한 설정이 `application.yml`에 추가되어야 한다면, 절대 값을 하드코딩하지 말고 아래 프로세스에 따라 `.env`로 분리해야 합니다.

### Step 1. 로컬 개발 및 테스트
1. `application.yml`에 필요한 구조를 추가합니다. (예: `secret-key: ${NEW_SECRET_KEY}`)
2. 본인 로컬의 `.env`에 `NEW_SECRET_KEY=실제_값`을 추가하고 정상 동작을 확인합니다.
3. **`.env.example` 파일에도 해당 키값 템플릿(값은 비운 채로)을 반드시 추가하고 커밋합니다.**

### Step 2. Notion 히스토리 등록 (Local용)
1. 노션 `.env (local용)`의 **'버전 기록'** DB에 새로운 버전을 추가합니다.
2. 어떤 키가 추가되었는지 변경 내용(Diff)을 명시하고, 본인의 새로운 `.env` 텍스트를 등록합니다.
3. 이때 상태는 '사용 중'이 아닌 '도입 예정'으로 설정합니다.

### Step 3. 개발계(Dev) 배포 준비 및 Secret 반영 🚨(가장 중요)
PR이 Approve 되어 **`dev` 브랜치에 머지(Merge) 되기 직전**에 다음 작업을 **직접** 수행해야 합니다.

1. 노션 `.env (dev용)` 버전 기록에 새로운 키값을 추가하여 버전을 생성합니다.
2. **GitHub Repository ➡️ Settings ➡️ Secrets and variables ➡️ Actions** 로 이동합니다.
3. **`ENV_DEV`** 시크릿을 편집하여 새로운 키값이 포함된 `.env` 텍스트로 업데이트합니다. (기존 내용에서 새 키값만 추가)
4. 디스코드 백엔드(`be`) 채널에 **"환경변수 `XXX`가 추가되었습니다. 로컬 `.env`를 업데이트해 주세요!"** 라고 전파합니다.
5. 이제 PR을 머지합니다! (배포 워크플로우가 새로운 Secret을 읽어 정상적으로 서버를 띄웁니다.)

### Step 4. 머지 후 상태 정리
1. 노션으로 돌아가 본인이 올렸던 '도입 예정' 버전들을 '사용 중'으로 변경합니다.
2. 기존에 '사용 중'이던 이전 버전은 '이전 버전' 태그로 변경하여 히스토리를 보존합니다.

---

## 4. 운영계(Prod) 환경변수 관리 규칙
운영 환경(`ENV_PROD`)은 잦은 변경을 막기 위해 PR 작성자 개인이 직접 수정하지 않습니다.
1. `main` 브랜치로 배포가 나가는 릴리즈 시점에 **백엔드 파트장**(배포 담당자)이 Dev 환경에 쌓인 추가된 환경변수들을 취합합니다.
2. 취합된 내용을 바탕으로 노션의 `.env (prod용)`을 갱신하고, GitHub Secret `ENV_PROD`를 업데이트한 뒤 Main 배포를 진행합니다.
